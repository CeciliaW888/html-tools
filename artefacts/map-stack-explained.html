The Map Stack

A map isn't one product — it's an assembly line. Data goes in one end; a finished map comes out the other — in a web app, a Python script, a dashboard, or a saved image. The same stations apply to all of them.

04Render — draw the map

The engine that turns tiles + data + style into a picture. Where it runs depends on you — a browser, a Python script, a BI tool, or a static image — but its job is the same everywhere.

Notice the pattern: Plotly, Folium and pydeck don't replace the web engines — they wrap them. So the renderer is the only layer that changes with your environment. Everything below it — data, tiles, style, the tile service — is identical whether you're in JavaScript, Python, or a BI tool.

⊞ Zoom in: the role of a tile service

Your map holds almost no data of its own. As it loads (and as you pan or zoom), the renderer keeps asking a tile service — a server whose only job is to hand out tiles — for just the squares it needs right now. The map is stitched together on demand. This is true in a browser, a Python script, or a BI visual alike.

That loop is a plain client–server relationship over HTTP — the same one your browser has with any website. The map is the client; the tile service is the server. Two consequences fall out of this:

The client
only ever asks. It pulls the tiles it needs and can't force a reply. It's also a client to several servers at once — tiles, style, fonts, geocoding, routing.
The server
can always refuse. It inspects every request and may say no — rate limits, referrer checks, missing key. A block isn't a broken connection; it's the server choosing not to answer.
So
whoever runs the server holds the power. The fixes are about that: an API key makes you a client it trusts; self-hosting makes you the server, so you stop asking permission.

Two jobs hide under the word "tiles": building them once (chop the data into tiles) and serving them live. The tile service is the serving half — and it can live in three places:

Provider's public server
Free, but rate-limited and often restricts who's allowed to call it.
Hosted provider
You get an API key; they run the server and scale it for you.
Your own server / storage
You host the tiles yourself. Full control, no outside limits.

The tile service is a live dependency, not a one-time export. If it's slow, blocks your app, or goes down, your map breaks — even when your own code is perfect. So picking the tile service is a real design decision, not a detail.

The one thing to remember

OpenStreetMap is the ingredients. MapLibre is the kitchen that plates them. Photon is the waiter who finds what you asked for. CARTO is the restaurant that does all three so you don't have to.

You almost never pick "one tool." You pick one option per layer — and you can swap any layer without touching the others.
