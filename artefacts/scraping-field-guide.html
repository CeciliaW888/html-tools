Chapter II — Locate Your Site

Three axes determine your stack.

Before picking a library, locate yourself in three dimensions. Every meaningful scraping decision is a point in this space — and most "should I use X or Y?" debates are arguments about the wrong axis.

→ Data Location

Is the data in initial HTML, rendered by JS, behind interaction, or wildly varied across sites? Determines whether you need a browser.

↑ Anti-Bot Pressure

Open sites need nothing. Friction sites need proxies and delays. Fortress sites (Cloudflare, Datadome) need residential IPs + stealth.

↗ Maintenance Appetite

Hand-wired control, framework defaults, or pay-to-skip. Pushes you up the build→buy ladder of tool categories regardless of the other two axes.

Building Blocks

HTTP + parser, hand-wired.
SPEED · fastest
COST · $0
STAGE · fetch + parse
When to use
  • Data is in the initial HTML response
  • You want full control over each request
  • You're scraping a single, well-structured site
When not to
  • Page needs JavaScript to populate
  • You'd reinvent retry / queue / dedupe logic
Top picks
httpxAsync HTTP, modern default
parselScrapy's parser, lxml speed + clean API
lxmlXPath support, very fast
selectolaxFastest HTML parser, CPU-bound jobs
→ Real code · JSON-LD first, selector fallback
Python 
import httpx, json
from parsel import Selector

r = httpx.get("https://example.com/product/123", timeout=10)
sel = Selector(text=r.text)

# Try JSON-LD first — free, structured, SEO-maintained
for script in sel.css('script[type="application/ld+json"]::text'):
    data = json.loads(script.get())
    if data.get("@type") == "Product":
        print(data["name"], data["offers"]["price"])
        break
else:
    # Fallback to selectors only if JSON-LD missing
    name = sel.css("h1.product-title::text").get()
    price = sel.css(".price::text").get()

from playwright.async_api import async_playwright

async with async_playwright() as p:
    browser = await p.chromium.launch()

    await page.wait_for_selector(".product-card")

    cards = await page.query_selector_all(".product-card")

Python 
import scrapy

class ProductSpider(scrapy.Spider):
    name = "products"
    start_urls = ["https://example.com/category"]

    def parse(self, response):
        for p in response.css(".product-card"):
            yield {
                "name":  p.css("h3::text").get(),
                "price": p.css(".price::text").get(),
                "url":   p.css("a::attr(href)").get(),
            }
        next_page = response.css("a.next::attr(href)").get()
        if next_page:
            yield response.follow(next_page, self.parse)

Chapter IV — Traps in Production

Eight traps that quietly break scrapers.

Each of these costs roughly a month of debugging to learn the hard way. Read them now and save yourself the tuition.

01

Reaching for a browser before checking for an API.

SYMPTOM · Playwright everywhere · slow + expensive

Open DevTools → Network → XHR. The page rendering with JavaScript is almost always calling a JSON endpoint internally. Hitting that endpoint with httpx is faster, cheaper, more stable, less detectable. Browsers should be the last resort, not the first reach.

02

Skipping JSON-LD entirely.

SYMPTOM · Hand-written selectors for fields already structured

Roughly 80% of e-commerce sites publish schema.org/Product as JSON-LD in the page head — the site owner maintains it for SEO. Free, structured, durable. Always try JSON-LD first. Selectors second. LLM extraction last.

03

Sync HTTP at scale.

SYMPTOM · 10-hour jobs that should be 6 minutes

A sync scraper spends 95% of its time blocked on network. Switching requests for httpx async and firing 100 concurrent calls is the single biggest performance win available, and the code is almost identical.

04

Vanilla Playwright on protected sites.

SYMPTOM · "Just a moment..." Cloudflare page · empty results

Default Chromium has tell-tale fingerprints (navigator.webdriver, missing plugins, CDP traces) that Datadome and Cloudflare detect in milliseconds. You need patchright, camoufox, or a managed stealth browser. And the trap runs deeper than the browser: plain HTTP clients are fingerprinted at the network layer too (TLS/JA4, HTTP/2), so a bare httpx call is just as visible — for direct API hits under pressure, reach for a TLS-impersonating client (curl_cffi). The deeper rule: anti-bot scores every layer together — TLS, headers, JS, IP, timezone — so a spoofed user-agent over a Python TLS fingerprint is more detectable, not less. Present one coherent identity across all layers, or none.

05

Datacentre proxies on serious sites.

SYMPTOM · Cheap proxies that get blocked instantly

AWS, GCP, DigitalOcean IPs are pre-flagged by every major anti-bot vendor. Residential and ISP proxies (Bright Data, Oxylabs, Smartproxy) cost $5–$15 per GB, and that's the line item you can't skip. Budget for it on day one, not month two.

06

Cancel-as-insert ghost rows.

SYMPTOM · Duplicate records · phantom inventory · "off-by-one" reports

Many source systems model cancellations or refunds as inserts of a new row instead of updates to the original. Naive incremental scrapers see both rows as real and inventory or counts double. Always check the source's update semantics before trusting any incremental refresh.

07

Treating LLM extraction as primary.

SYMPTOM · $2k/month bills for what selectors do for $0

LLMs are seconds-per-page and cents-per-call. Selectors are microseconds and free. The right pattern: deterministic methods first, LLM only on novelty. Even better — capture the selector path the LLM implicitly used and reuse it next time. Self-healing selectors collapse your cost.

08

Anti-bot drift.

SYMPTOM · Scraper that worked in March returns empty pages in June

Stealth patches age. Fingerprint surfaces shift. Detection vendors update weekly. A scraper that ran clean in spring will silently start returning empty pages in summer. Build monitoring (success rate per domain, response-size distribution, captcha-encounter rate) into the pipeline from day one. Drift is the rule, not the exception.

Chapter V — Find Your Data · Extracting Fields

Five ways to extract fields from a page.

This is the Extract ladder — pulling fields out of a page you already have. (Finding the page's URL in the first place is a different problem — Discover, the next chapter.) Try these in order; each that works lets you skip everything below. Most scrapers never pass step three.

Try in order → 01Network / XHR → 02JSON-LD → 03CSS Selectors → 04XPath → 05ARIA Locators
Try 01 First

Network / XHR

Before writing any parsing code, check whether the page fetches its data from a hidden JSON API. If it does, you can call that endpoint directly — no HTML, no selectors, no browser needed. This is almost always the cleanest data you'll find.

How to spot it
  1. Open DevTools → Network tab
  2. Filter to Fetch / XHR
  3. Reload — watch for JSON responses
  4. Click a request → Response tab to inspect it
  5. If it has your data: right-click → Copy as cURL
Use when
  • Fetch/XHR calls appear in DevTools
  • Response is structured JSON
  • Pagination is a query parameter
Skip when
  • No XHR visible — data is server-rendered HTML
  • Endpoint requires rotating session tokens
Call the endpoint directly (no browser)
Python 
import httpx

# Copy headers from DevTools → request headers
r = httpx.get(
    "https://example.com/api/v2/products",
    params={"page": 1, "limit": 50},
    headers={
        "accept": "application/json",
        "x-requested-with": "XMLHttpRequest",
    },
)
for item in r.json()["data"]:
    print(item["name"], item["price"])
Or capture responses inside Playwright
Python 
captured = []

async def on_response(resp):
    if "/api/products" in resp.url and resp.status == 200:
        try: captured.append(await resp.json())
        except Exception: pass

page.on("response", on_response)
await page.goto("https://example.com/shop")
await page.wait_for_load_state("networkidle")
print(captured)
Try 02 Second

JSON-LD

Many sites — especially e-commerce — embed perfectly structured data in a <script type="application/ld+json"> tag for search engines. That same data is free for you to read. No selector needed — just json.loads().

How to spot it

DevTools → Elements → search ld+json. Look for a JSON object with "@type": "Product", "Article", "Recipe", etc.

Common schema.org types
TypeTypical pageKey fields
Producte-commerce PDPname, offers.price, sku
Articlenews / blogheadline, author, datePublished
Recipefood sitename, recipeIngredient, cookTime
BreadcrumbListanyitemListElement[].name
Eventevent listingsname, startDate, location
Why it outlasts selectors
  • Structured — no guessing element positions
  • SEO-maintained by site owner, stays accurate
  • Survives CSS and layout redesigns
Extraction pattern (parsel)
Python 
import httpx, json
from parsel import Selector

r = httpx.get("https://example.com/product/42")
sel = Selector(text=r.text)

for blob in sel.css(
    'script[type="application/ld+json"]::text'
).getall():
    data = json.loads(blob)
    if data.get("@type") == "Product":
        name  = data["name"]
        price = data["offers"]["price"]
        sku   = data.get("sku", "")
        print(name, price, sku)
        break
Try 03 Third

CSS Selectors

The workhorse of web scraping. CSS selectors target HTML elements by tag, class, ID, or attribute. They work on static HTML without a browser — fast and cheap. Most scrapers live here.

Core syntax
SelectorMatches

.cardclass includes "card"
#mainid="main"
[data-id="42"]data-id equals "42"
div.card<div> that also has class card
ul > liDirect <li> children of <ul>
h2 + p<p> immediately after <h2>
:nth-child(2n)Every even child
parsel extras (not standard CSS)
SelectorReturns
h1::textText content of <h1>
a::attr(href)Value of href attribute
parsel example
Python 
import httpx
from parsel import Selector

r = httpx.get("https://example.com/products")
sel = Selector(text=r.text)

for card in sel.css(".product-card"):
    name  = card.css("h3::text").get()
    price = card.css(".price::text").get()
    url   = card.css("a::attr(href)").get()
    print(name, price, url)
Playwright CSS extras
Python 
# Playwright adds :has-text(), :visible, :has()
in_stock = page.locator(
    ".product-card:has-text('In stock'):visible"
)
print(await in_stock.count())

# Attribute selectors — more stable than generated class names
items = page.locator("[data-product-id]")
async for item in items.all():
    pid = await item.get_attribute("data-product-id")
    print(pid)
Avoid class names like sc-a7f3b — they change every deploy. Prefer [data-*] attributes or structural selectors.
Try 04 Fourth

XPath

XPath is a full query language for HTML trees. Reach for it when CSS falls short: selecting elements by text content, traversing up to a parent, or picking table cells by their header name.

Syntax reference
ExpressionMeaning
//divAny <div> anywhere in the document
@hrefThe href attribute value
text()Direct text nodes of the element
[@class='card']Exact class attribute match
[contains(@class,'card')]Class contains "card"
[contains(text(),'Price')]Text contains "Price"
[1] / [last()]First / last match
parent::divNearest <div> parent
ancestor::article[1]Nearest <article> ancestor
following-sibling::dd[1]First <dd> after this node
When CSS can't, XPath can
  • Selecting a parent or ancestor element
  • Matching by visible text (text()='...')
  • Table cells selected by header name
parsel example
Python 
from parsel import Selector

sel = Selector(text=html)

# Select by text — CSS can't do this natively
price = sel.xpath(
    '//dt[contains(text(),"Price")]'
    '/following-sibling::dd[1]/text()'
).get()

# Walk UP to the containing card
sold_out = sel.xpath(
    '//span[text()="Sold Out"]/ancestor::article[1]'
)
for card in sold_out:
    print(card.xpath('.//h3/text()').get())

# Attribute extraction
hrefs = sel.xpath('//a[@rel="nofollow"]/@href').getall()
Playwright XPath
Python 
# Prefix with "xpath=" in Playwright locators
heading = await page.locator("xpath=//h1").inner_text()

# Table cell selected by header text
cell = page.locator(
    "xpath=//th[text()='Price']/following-sibling::td[1]"
)
print(await cell.inner_text())
Try 05 Playwright

ARIA & Role Locators

Playwright's get_by_* methods find elements by what they mean — role, label, visible text, or placeholder — not how they look. They survive CSS refactors and are Playwright's recommended default for form interactions and navigation.

get_by_* methods
MethodFinds by
get_by_role("button", name="Submit")ARIA role + accessible name
get_by_label("Email")<label> text or aria-label attr
get_by_text("Add to cart")Any visible text content
get_by_placeholder("Search...")Input placeholder
get_by_alt_text("Logo")Image alt attribute
get_by_test_id("submit-btn")data-testid attribute
Locator precedence rule
  • 1st: get_by_role — most semantically stable
  • 2nd: get_by_label / get_by_text
  • 3rd: get_by_test_id — needs eng cooperation
  • Last: CSS / XPath — only when above won't work
Login + navigation example
Python 
from playwright.async_api import async_playwright

async with async_playwright() as p:
    page = await (
        await p.chromium.launch()
    ).new_page()
    await page.goto("https://example.com/login")

    # Form — survives CSS refactors
    await page.get_by_label("Email").fill("user@example.com")
    await page.get_by_label("Password").fill("secret")
    await page.get_by_role(
        "button", name="Sign in"
    ).click()

    await page.wait_for_url("**/dashboard")
    h1 = await page.get_by_role(
        "heading", level=1
    ).inner_text()
    print(h1)

    # Nav links — chain locators
    nav = page.get_by_role("navigation")
    for link in await nav.get_by_role("link").all():
        print(await link.inner_text())
Chapter VI — Discover · Finding the URL

You can't fetch a URL you don't have.

Most guides start at Fetch and assume a URL in hand. Often you don't have one — you have an identity (a name, an address, a registration number) and must find the site first. That's Discover, pipeline stage 1, and it's where enrichment projects actually begin.

From identity → website — three hops
1Resolve a clean identity
name + address + state. A business register (e.g. ABN Lookup) fills gaps and canonicalises the name — it returns identity + location, not a website.
2Find a candidate site
search name + address + state — a maps/places API returns the official site directly (its website field); a web-search API gives the top organic result. Search the name + location, never the registration number — a number has nothing to match on.
3Verify it's the right one
does the page mention the name and match the suburb / postcode? A multi-signal match is the accept test — and the step people skip, where wrong data enters.
When the business is gone — a trust-ordered fallback

A defunct business may have no live site. Don't fail — climb a fallback chain, and tag how far down you had to go as provenance.

1Its own site, still live?
→ use ithighest trust
2An archived snapshot?
→ fetch the old homepage from a web archive (Wayback Machine)high
3A parent company?
→ resolve the parent's site and infer from itlower · holding-co risk
↳ a diversified parent's industry ≠ the subsidiary's — mark provenance "inferred from parent"
4Nothing left?
→ infer from name + address text alonelowest · flag it
Stack note: Discover is search/API-heavy on open sites — it lives on the cheapest spine rung (a places/search API + httpx + a parser). No browser or stealth needed. Cache results per entity; a false match is the costly failure, so verification (hop 3) is not optional.
